背景：Kiro 的崛起與承諾

Kiro 於 2025 年 7 月由 AWS 推出，是基於 Code OSS 的 AI 驅動 IDE，以「Spec-Driven Development」為核心差異化特色，上線後迅速吸引數十萬開發者加入等候名單。

「初始定價看起來合理：免費版 50 次互動、Pro $19/月 1,000 次互動、Pro+ $39/月 3,000 次互動，額外互動每次 $0.04。」 — The Register, 2025 年 8 月報導（內容經改寫）

初始承諾的關鍵特點

  • 一次「互動」可能包含 3–5 分鐘的持續 AI 迭代工作
  • 計費單位簡單明瞭：一個問題 = 一次互動
  • 開發者可以預測每月成本

五大核心批評

💸

1. 成本不可預測性

從固定「一問一答」變為依複雜度浮動計費，開發者無法事先估算月度支出。分數 credit 制（0.01 增量）讓計算更加困難。

🔀

2. 雙軌制造成混亂

Spec 請求與 Vibe 請求的區分不直覺，且系統會自動將「過於複雜」的 Vibe 請求導向 Spec 請求，消耗更昂貴的配額。

📉

3. 實際額度大幅縮水

表面價格僅微調（$19→$20），但可用互動次數從 1,000 驟降至 350，等效漲價超過 180%。

🐛

4. 計費 Bug 侵蝕信任

上線即出現計費錯誤，單一任務消耗多倍請求。AWS 事後才承認 Bug 存在，但信任已受損。

🚪

5. 免費版形同虛設

免費版完全移除 Spec 請求，等於 Kiro 最核心的差異化功能（Spec-Driven Development）對免費用戶關閉大門。

結論與建議

核心問題

  • ▸ 從簡單可預測的互動制，轉向複雜不透明的 Token/Credit 制
  • ▸ 實質大幅漲價卻包裝為「計費優化」
  • ▸ 計費 Bug 暴露系統成熟度不足
  • ▸ 免費版閹割核心功能，阻礙社群成長

對 Kiro 的建議

  • ▸ 回歸簡單透明的計費模型，降低認知負擔
  • ▸ 提供即時用量儀表板與成本預警機制
  • ▸ 恢復免費版的 Spec 功能存取（即使有限）
  • ▸ 建立明確的定價承諾，避免再次「先甜後苦」
